# Native Components

A **native UI component** exposes a platform view (a `UIView` subclass on iOS, a `ViewGroup` subclass on Android) as a JSX element. It is how libraries such as `react-native-maps`, `react-native-webview`, and `react-native-video` render: the JS side declares props and children, while the native side owns a real view in the hierarchy. Under the Old Architecture, ViewManagers bridge props and events; under Fabric, native components have typed component descriptors, C++ shadow nodes, and direct Fabric mounting.

## Old Arch: ViewManager

On iOS, subclass `RCTViewManager`, override `view()` to return your `UIView`, and export props with `RCT_EXPORT_VIEW_PROPERTY`. On Android, subclass `SimpleViewManager<T>` or `ViewGroupManager<T>`, override `createViewInstance`, and expose props with `@ReactProp`. Every prop change goes through the bridge, typed as `int`, `float`, `String`, `ReadableArray`, `ReadableMap`, or `Dynamic`. The bridge batches updates per frame.

## New Arch: Fabric Native Components

A **Fabric component** is defined by a spec (TypeScript for Codegen), a C++ `ShadowNode` class, and a per-platform component. Props are strongly typed, and the shadow tree is built and diffed in C++ off the main thread. Mount and update operations land on the main thread as a pre-committed diff, which is faster and more predictable than the Old Architecture's per-prop bridge calls. Commands (the imperative API) flow through the same Codegen-generated path as methods.

## Component Descriptors and Shadow Nodes

A **component descriptor** names the component and ties together its `ShadowNode` class, its `Props` struct, and its `EventEmitter` type. The `ShadowNode` owns the props and state; it is an immutable, copy-on-write C++ object. On each render, Fabric diffs the old and new shadow trees, computes mutation instructions, and applies them on the main thread. Custom `ShadowNode`s are required only when you need custom layout or off-main-thread state.

## Props and Their Marshalling

Under the Old Architecture, props flow as an `NSDictionary` / `ReadableMap`, and type mismatches fail silently. Under the New Architecture, props are typed C++ structs generated by Codegen from your TS spec, with compile-time checking; default values, enums, and nested objects are handled by the generator. Adding a prop after release requires backward-compatibility handling — the native side must tolerate older JS that does not send the prop.

## Events from Native to JS

Old Architecture: attach an event via `RCTDirectEventBlock` (iOS) or `RCTEventEmitter.receiveEvent` (Android); the JS side receives a function prop `onSomething`. New Architecture: Fabric's `EventEmitter` provides strongly typed event dispatch tied to the specific component instance, and Codegen generates the emitter. The `bubbles`/`captures` behavior matches DOM semantics and is configurable. Custom event types need a spec entry plus the emitter implementation.

## Commands (Imperative API)

**Commands** let you call methods on a specific native view instance — `scrollTo`, `focus`, `play`. Old Architecture: `RCTUIManager dispatchViewManagerCommand:commandID:args:` on iOS, `UIManagerModule.dispatchViewManagerCommand` on Android. New Architecture: Codegen produces a ref-based API (`ref.current.scroll(100)`) backed by a `ViewCommand` `ShadowNode` mutation. Commands are asynchronous under the Old Architecture and synchronous under the New.

## Measurement (Yoga vs Custom)

Most components defer layout to **Yoga**, RN's Flexbox engine. Some components have intrinsic sizes the layout engine cannot know without inspecting content: text (measured per glyph), images (measured post-decode), video (aspect-ratio-derived). Old Architecture: override `measure` on the ShadowView to provide layout. New Architecture: use a custom `ShadowNode` plus a `LayoutableShadowNode::layout` override. Getting this right is why `<Text>` and `<Image>` have complex layout code.

## Accessibility

Propagate `accessibilityLabel`, `accessibilityRole`, `accessibilityHint`, and `accessibilityState` from JS to the platform accessibility API (`UIAccessibility` / `AccessibilityNodeInfo`). Expose focusability, custom actions, and live-region updates. Fabric components receive accessibility props via Codegen, but you still have to wire them to the platform view. TalkBack and VoiceOver expose bugs that sighted testing misses — test both on every component release.

## iOS Specifics (UIView Subclass)

Your view subclass must tolerate being reconfigured mid-lifecycle, because RN may apply props in any order, before or after `didMoveToWindow`. Auto Layout inside an RN-managed view is allowed but pointless (RN sets the frame directly); use it only if the subview hierarchy is self-contained. `layoutSubviews` is the right place for view-internal layout decisions. Pay attention to `setNeedsDisplay` versus `setNeedsLayout` — RN's commit is already layout, not draw.

## Android Specifics (ViewGroup Subclass)

`onMeasure` and `onLayout` are owned by RN's layout system — overriding them usually breaks the shadow tree. Use `requestLayout()` cautiously, as it can cause re-measurement of the whole tree. For custom drawing, override `onDraw` on a regular `View`. If you subclass `ViewGroup`, remember that Android recycles views aggressively, so reset state in `onAttachedToWindow` rather than the constructor.

## State and Re-measurement

Under Fabric, a `ShadowNode` can hold state separate from props — useful for async-measured content (image dimensions after decode, text layout). State updates trigger a re-measure and commit. The Old Architecture equivalent is `ShadowView.setIntrinsicSize`, which is less powerful and not always thread-safe. Keep native-computed state minimal; every update is a diff pass.

## View Recycling (Android)

Android's RecyclerView pattern is not automatic for RN components, but the UI manager does reuse native view instances across reconciliation. Your component must tolerate being bound to new props without leaked state from the previous binding — reset scroll position, cancel in-flight animations, and clear listeners on each `updateProperties` call.

## Integration with Reanimated and Gesture Handler

Reanimated wraps native views to intercept style props for worklet-driven updates; custom components that do not forward `ref` properly break this. `react-native-gesture-handler` v2 attaches gesture handlers to native views and expects standard touch forwarding. When building a custom component, expose the underlying native view via `forwardRef` and respect the touch responder chain.

## Codegen for Fabric Components

A component spec (`MyViewNativeComponent.ts`) exports a `codegenNativeComponent<Props>(name)`. Running Codegen produces C++ `ShadowNode` headers, Objective-C props structs, Java props classes, and a registry stub. Build tooling plugs it into Xcode and Gradle automatically under the New Architecture template. Stay on the generated output — hand-writing what Codegen emits is a maintenance trap.

## Testing Native Components

Unit-test the native view in isolation with XCTest/Espresso, the same as any platform view. Integration tests benefit from snapshot testing at the RN level: render the component in a test app, capture the native view tree, and compare across versions. Detox can drive gestures on custom components via accessibility identifiers.

## Third-Party Libraries as Case Studies

`react-native-maps` is the canonical complex native component — platform SDK, markers as nested components, gesture delegation, callback events. `react-native-webview` demonstrates event coordination between JS and native content. `@shopify/flash-list` shows custom layout managers. Reading their source is the fastest way to learn idiomatic native-component patterns.

## Performance Considerations

Creating and destroying native views is expensive — recycle instead of unmounting. Prop updates are diffed per key, but large props (long strings, nested objects) still cost marshalling time. Under Fabric, minimize shadow tree depth and prefer primitive props over nested structs. Measure native view init time with Xcode Instruments (Time Profiler on the main thread) and Android's Choreographer frame stats.
